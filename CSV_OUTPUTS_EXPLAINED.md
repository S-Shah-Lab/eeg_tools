# CSV outputs in the motor-imagery analysis package

This document explains every CSV file generated by the current package, how each table is built, what each field represents, and the exact line of code that writes the information to that CSV.

The current output layout is:

```text
<save-path>/<input-stem>/
├── images/
├── csv/
│   ├── analysis_parameters.csv
│   ├── paradigm_validation_summary.csv
│   ├── trial_quality_summary.csv
│   ├── psd_mean_per_channel_per_bin.csv
│   ├── psd_long_format.csv
│   ├── signed_r2_per_band_per_channel.csv
│   ├── stat_summary_per_band.csv
│   ├── bootstrap_stats_per_band.csv
│   ├── erd_ers_timecourse.csv
│   └── decoding_summary.csv
├── report_metadata.json
├── <input-stem>_cleaned_raw.fif
└── motor_imagery_report.pdf
```

# Main motor-imagery CSV files

## 1. `analysis_parameters.csv`
### What it represents
This file records the run configuration and basic raw-data properties used for the analysis.
It is mainly useful for reproducibility: it tells you what parameters were used when the report and CSV files were generated.

### Columns
| Column | Meaning |
|---|---|
| `parameter` | Name of the analysis setting or raw-data property |
| `value` | Value used in this run |

---

## 2. `bootstrap_stats_per_band.csv`
### What it represents
This is a more focused bootstrap-only table per comparison and frequency band.
It stores the observed statistic, bootstrap distribution mean, percentile confidence interval, one-sided p-value, Monte-Carlo p-value interval, and simulation count.
It overlaps with information stored in `stat_summary_per_band.csv`, but keeps the bootstrap details together in a simpler table.

### How the information is generated
Bootstrap results are produced by `_bootstrap_test()`.
The bootstrap resamples rows with replacement and recomputes the lateralized statistic on each resample.

### Columns
| Column | Meaning |
|---|---|
| `comparison` | Task/rest comparison |
| `band` | Frequency band |
| `observed` | Observed lateralized statistic (positive is good)|
| `boot_mean` | Mean of the bootstrap statistic distribution |
| `ci_low` | Lower percentile confidence bound |
| `ci_high` | Upper percentile confidence bound |
| `ci_level` | Confidence interval level, currently `95.0` by default |
| `p_value` | One-sided bootstrap p-value for the configured null direction |
| `p_ci_low` | Lower Monte-Carlo uncertainty bound for `p_value` |
| `p_ci_high` | Upper Monte-Carlo uncertainty bound for `p_value` |
| `n_simulations` | Number of bootstrap resamples, currently expected to be `9999` unless overridden |

## 5. `paradigm_validation_summary.csv`
### What it represents
This file checks whether the expected motor-imagery event labels were found in the recording.
The expected labels are:
- `left`
- `right`
- `left_rest`
- `right_rest`
It also reports whether the event sequence can be split into complete four-event blocks.

### Columns
| Column | Meaning |
|---|---|
| `label` | Expected or unexpected event label |
| `count` | Number of events found with that label |
| `status` | `ok`, `missing`, `unexpected`, or `remainder_<n>` |

---

## 6. `psd_long_format.csv`
### What it represents
This file contains the same PSD family of information in long format.
It is a larger database that could be used for plotting, filtering, grouping, and external statistics.
Each row is one `(trial-epoch, condition, channel, frequency)` measurement.

To clarify the naming: 
- "left" is for trial type --> The instruction to move a left hand was given
- "rest_left" is for trial type --> The instruction to relax after a left hand movement was given
- "left_1" identifies the first out of 8 `left` trials
- For each one, aa PSD value is given per channel per bin

### How the information is generated
The export loops over every PSD batch in `self.psds_dict`, identifies its base condition, then writes one row per motor-imagery channel and frequency bin.
It also computes a decibel version of PSD power.

### Columns
| Column | Meaning |
|---|---|
| `batch` | Epoch-batch key, such as `left_1` or `right_rest_2` |
| `condition` | Base condition inferred from `batch` |
| `channel` | Motor-imagery channel name |
| `frequency_hz` | Frequency bin in Hz |
| `psd_power` | Raw PSD power |
| `psd_db` | PSD converted to dB as `10 * log10(psd_power * 1e12)` |

## 7. `psd_mean_per_channel_per_bin.csv`
### What it represents
This file gives the average power spectral density for each motor-imagery channel and each frequency bin.
It is a wide-format PSD table: one row per motor-imagery channel, one column per frequency bin.
The values are raw Welch PSD values, I think it should be `V²/Hz`.

Note that this file condenses a lot of information found in `psd_long_format.csv`, e.g. all trials are condensed into a single value per channel per frequency bin

### How the information is generated
PSD values are first computed for each epoch batch in `_generate_psds()`.
For each batch, the code keeps both per-epoch PSDs and the mean PSD across epochs.

### Columns
| Column | Meaning |
|---|---|
| `channel` | Motor-imagery channel name |
| Frequency-bin columns, for example `1`, `1.5`, `2` | Mean PSD power at that frequency bin |

## 8. `signed_r2_per_band_per_channel.csv`
### What it represents
This file contains the signed effect size per comparison, frequency band, and motor-imagery channel.
The effect is signed `r²`, computed from the correlation coefficient as `corr * abs(corr)`

A signed value is useful because ordinary `r²` loses the direction of the effect, we care about the direction because we care specifically about desincronization (power goes down compared to rest). Power going up compared to rest is not a sign of typical motor cortex activity as the one we expect for hands movement.

In this table:
- positive and negative signs preserve task/control directionality from the treatment coding
- values closer to 0 indicate weaker task/rest separation for that channel and band
- larger absolute values indicate stronger task/rest separation

Note that all `rest` and `movement` trials for a given hands are used 
A correlation coefficient is formed using (maximum, based on quality) 8 + 8 data points, with either y = 0 or 1 based on class

### Columns
| Column | Meaning |
|---|---|
| `comparison` | Task/rest comparison, usually `left_vs_left_rest` or `right_vs_right_rest` |
| `band` | Frequency band, for example `8-13Hz` |
| One column per motor-imagery channel | Signed `r²` value for that channel |

## 9. `stat_summary_per_band.csv`
### What it represents
This is the compact statistical summary for each task/rest comparison and frequency band.
It combines the observed lateralized statistic, bootstrap confidence interval, bootstrap p-value, permutation p-value, FDR-corrected p-values, and sample counts.
Because this function currently returns `ipsilateral - contralateral`, a positive `observed_delta` means the summed ipsilateral effect is greater than the summed contralateral effect (this is what we want, by definition we kept a positive overall effect our signature).
The file also includes a text `direction` column generated by `_effect_direction_label()`.

### Columns
| Column | Meaning |
|---|---|
| `comparison` | Task/rest comparison |
| `band` | Frequency band |
| `observed_delta` | Observed lateralized statistic, currently `sum(ipsilateral effects) - sum(contralateral effects)` |
| `direction` | Human-readable sign interpretation of `observed_delta` |
| `bootstrap_ci_low` | Lower percentile of the 95% bootstrap distribution |
| `bootstrap_ci_high` | Upper percentile of the 95% bootstrap distribution |
| `bootstrap_p` | One-sided bootstrap p-value |
| `bootstrap_p_fdr` | Benjamini-Hochberg FDR-corrected bootstrap p-value |
| `permutation_p` | One-sided approximate permutation p-value |
| `permutation_p_fdr` | Benjamini-Hochberg FDR-corrected permutation p-value |
| `neg_ln_bootstrap_p` | `-ln(bootstrap_p)` style significance transform |
| `neg_ln_bootstrap_p_fdr` | `-ln(bootstrap_p_fdr)` style significance transform |
| `n_task_blocks` | Number of task blocks contributing to the comparison |
| `n_control_blocks` | Number of control/rest blocks contributing to the comparison |
| `n_simulations` | Number of bootstrap simulations used, currently expected to be `9999` unless overridden |

## 10. `trial_quality_summary.csv`
### What it represents
This file summarizes how many epochs were available and how many survived epoch rejection (due to bad quality) for each condition (at most 8 trials per type, 6 epochs per trial = 48 total epochs per type)
It helps diagnose whether too many trials were dropped.


### Columns
| Column | Meaning |
|---|---|
| `condition` | `left`, `right`, `left_rest`, `right_rest`, `other`, or `total` |
| `candidate_epochs` | Number of epoch candidates found from annotations/events |
| `kept_epochs` | Number of epochs retained in `self.epochs_dict` |
| `dropped_epochs` | `candidate_epochs - kept_epochs`, floored at 0 |
| `dropped_pct` | Percent of candidate epochs that were dropped |

## 3. `decoding_summary.csv`
### What it represents
This file stores optional task-vs-rest decoding metrics.
The decoder asks a simple question:
> Given bandpower features from one ROI, can a classifier distinguish task blocks from rest blocks?
The classifier is a shrinkage Linear Discriminant Analysis model inside a pipeline with standard scaling.
Decoding is run separately for:
- each task/rest comparison
- each frequency band
- `contra` ROI
- `ipsi` ROI


## 4. `erd_ers_timecourse.csv`
### What it represents
This file stores the within-trial ERD/ERS time course for common motor-imagery bands.
It shows how task power differs from matched rest power across time segments inside a trial.
Current exported bands:
- `8-13Hz`, usually treated as the mu band
- `13-31Hz`, usually treated as the beta band
Current exported ROI:
- `contra` only

## One-sided test direction
The statistical code currently computes the lateralized statistic as:

```text
sum(ipsilateral effects) - sum(contralateral effects)
```

That is the value exported as `observed_delta` in `stat_summary_per_band.csv` and as `observed` in `bootstrap_stats_per_band.csv`.

So, in the current code:

- `observed_delta > 0` means ipsilateral summed effect is greater than contralateral summed effect
- `observed_delta < 0` means contralateral summed effect is greater than ipsilateral summed effect

This matters because your visual interpretation says “good” means `Delta > 0`.
The CSV preserves that sign directly, so downstream scripts can use the same rule:

```text
good region: observed_delta > 0
bad/opposite region: observed_delta <= 0
```

## FDR correction

`stat_summary_per_band.csv` includes FDR-corrected bootstrap and permutation p-values.
The correction is applied after the raw band-wise p-values are generated.
The corrected fields are:

- `bootstrap_p_fdr`
- `permutation_p_fdr`
- `neg_ln_bootstrap_p_fdr`

## PSD wide vs PSD long

The two PSD CSVs intentionally duplicate some information in two different shapes:

- `psd_mean_per_channel_per_bin.csv` is compact and convenient for quick inspection
- `psd_long_format.csv` is better for plotting, grouping, filtering, and external analysis in pandas/R

## Decoding metrics

`decoding_summary.csv` should not be interpreted as proof that motor imagery is physiologically meaningful by itself.
It is a classifier-based diagnostic:

- strong decoding suggests task/rest feature separation in the selected band and ROI
- weak decoding suggests limited separability, noisy features, few trials, or no robust task/rest effect
- compare `contra` and `ipsi` rows to inspect lateralized separability
